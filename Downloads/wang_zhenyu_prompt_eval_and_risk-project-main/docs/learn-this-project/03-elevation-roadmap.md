# 03 — Elevation Roadmap

## Executive view (top 5 priorities)

1. **Complete J2–J5 judges** — only J1 is implemented; the full five-judge pipeline is the core promise of the framework. Without J2–J5, the meta-judge aggregator can't run.
2. **CI/CD integration** — the framework has no GitHub Actions gate that runs prompts through the judges on PR. Without this, prompt security is still manual.
3. **Remove hardcoded AWS profile** — `wang_zhenyu_dev` in `one_03_boto_ses.py` prevents any other developer or CI agent from running the code without surgery.
4. **Integration test layer** — unit tests cover `evaluations.py` and `llm_output.py` but the `tests_int/` directory is empty; no automated test exercises a real Bedrock call.
5. **Expand to UC2–UC6** — only UC1 (3 of 5 steps) is implemented; the five other use cases in the catalog exist only as specs.

## Upgrade areas

### 1. Judge completeness (J2–J5)

**Current state**: `prompt_risk/judges/` contains only `j1_over_permissive.py`. J2 (sensitive data), J3 (role confusion), J4 (instruction conflict), J5 (logic ambiguity) are defined in `docs/source/03-Judge-Catalog/index.rst` and `docs/source/07-Judge-Design/index.rst` but have no implementation.

**Senior-engineer target state**: All five judges implemented following J1's pattern. A `run_all_judges()` orchestrator calls them in parallel, collects results, and runs a meta-judge that produces a single aggregated risk report. The meta-judge applies use-case-specific weights (different use cases have different risk tolerance profiles).

**Alternatives**:
- *Single "mega-judge" prompt* — one LLM call evaluating all five dimensions simultaneously. Pro: cheaper (one call). Con: LLMs perform worse on multi-objective tasks; harder to isolate which dimension fired; model confidence isn't granular.
- *Deterministic checks for J2 (secrets)* — use a library like `detect-secrets` or `trufflehog` for J2 instead of an LLM judge. Pro: fast, zero cost, no false negatives on known patterns. Con: misses novel/obfuscated secrets.

**Knowledge prerequisites**: LLM prompt engineering (few-shot evaluation prompts), Pydantic structured output design, parallel async calls (asyncio or ThreadPoolExecutor).

**Suggested learning path**:
1. Anthropic's "Evals" guide (prompt engineering for evaluation)
2. OpenAI Evals framework — different tech stack, same concepts
3. Pydantic v2 validators deep-dive
4. Python `concurrent.futures.ThreadPoolExecutor` for parallel LLM calls

### 2. CI/CD prompt security gate

**Current state**: `.github/workflows/main.yml` runs unit tests (`mise test`) only. No job runs prompts through judges on PR. Prompt version changes in `data/` are merged without automated security review.

**Senior-engineer target state**: A `prompt-security` CI job triggers on any change to `data/**/versions/**`. It runs the relevant judge(s) on the changed prompt version(s), fails the PR if any judge scores CRITICAL or HIGH, and posts a summary comment.

**Alternatives**:
- *Pre-commit hook* — run judges locally before commit. Pro: fast feedback. Con: requires AWS credentials locally, expensive, slows down every commit.
- *Scheduled scan* — run judges nightly on all prompt versions. Pro: no per-PR cost. Con: vulnerable prompts merge before detection.
- *Manual PR review* — humans review prompt changes. Pro: free. Con: doesn't scale; humans miss subtle authorization creep.

**Knowledge prerequisites**: GitHub Actions workflow authoring, GitHub Actions OIDC for AWS authentication (no stored secrets), PR comment API via `gh` CLI.

**Suggested learning path**:
1. GitHub Actions: `paths:` filters and `needs:` job dependencies
2. AWS IAM OIDC for GitHub Actions (official AWS guide)
3. `gh pr comment` CLI usage

### 3. Multi-developer / CI credentials

**Current state**: `prompt_risk/one/one_03_boto_ses.py` hardcodes `profile_name="wang_zhenyu_dev"`. This means the package only works for one specific developer's local setup.

**Senior-engineer target state**: Credentials resolved from environment in priority order: (1) `AWS_PROFILE` env var, (2) `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` env vars (for CI), (3) instance metadata (for EC2/Lambda), (4) fallback to a configurable default profile. The profile name is never hardcoded.

**Alternatives**:
- *Pass the boto3 client explicitly* — already partially done (runners accept `client` as a parameter). Could remove the `one` singleton entirely and require callers to inject their own client. Pro: maximally flexible. Con: more boilerplate at call sites.
- *AWS IAM Roles Anywhere / SSO* — federated credentials with short-lived tokens. Pro: production-grade. Con: heavy setup for a research project.

**Knowledge prerequisites**: boto3 credential chain, `os.environ`, AWS IAM basics.

**Suggested learning path**:
1. boto3 credentials configuration guide (official docs)
2. 12-factor app configuration principles

### 4. Integration test layer

**Current state**: `tests_int/` directory exists but is empty. No test exercises a real Bedrock call, verifies that `run_p1_extraction()` produces valid output end-to-end, or checks that a J1 judge correctly scores a known-vulnerable prompt.

**Senior-engineer target state**: A suite of integration tests (tagged `@pytest.mark.integration`) that:
- Run each implemented prompt (P1, P2, P3, J1) on one normal and one attack case.
- Assert the output passes Pydantic validation.
- Assert attack cases score CRITICAL or HIGH on J1 (known-answer testing for the judge).
- Skipped in CI unless `RUN_INTEGRATION_TESTS=1` env var is set and AWS credentials are present.

**Alternatives**:
- *Bedrock mock (botocore stubber)* — mock the Bedrock API at the boto3 layer. Pro: fast, no cost, runs in CI without credentials. Con: doesn't test the real model's behavior; misses prompt-regression bugs.
- *Recorded cassettes (VCR pattern)* — record real API responses once, replay in CI. Pro: real responses, deterministic. Con: expires when model version changes; storage overhead.

**Knowledge prerequisites**: `pytest.mark` for test classification, `botocore.stub.Stubber`, `pytest-recording` (cassette pattern).

**Suggested learning path**:
1. pytest fixtures and marks (official docs)
2. botocore Stubber documentation
3. `pytest-recording` / `vcrpy` README

### 5. Use case expansion (UC2–UC6)

**Current state**: UC1 (claim intake) has P1–P3 implemented plus J1 security evaluation. UC2–UC6 are fully specified in `docs/source/02-Use-Case-Catalog/index.rst` but have no implementation. P4 (coverage check) and P5 (routing) for UC1 also have empty data directories.

**Senior-engineer target state**: At minimum UC2 (underwriting knowledge assistant — RAG) and UC6 (policyholder self-service chatbot) implemented. These represent opposite ends of the risk spectrum (internal tool vs. public-facing chatbot) and would exercise different prompt security profiles. Each use case should have its own judge weights in the meta-judge.

**Alternatives**:
- *Focus depth over breadth* — fully instrument UC1 (all 5 steps + all 5 judges + meta-judge) before adding new use cases. Pro: one complete vertical is more useful for a demo than six shallow ones. Con: misses the multi-use-case risk weighting story.
- *Community-contributed use cases* — publish the spec and accept PRs. Pro: scalable. Con: quality control overhead.

**Knowledge prerequisites**: RAG architecture (for UC2), tool-use / function-calling API (for UC3–UC5), multi-turn conversation management.

**Suggested learning path**:
1. AWS Bedrock Knowledge Bases for RAG
2. Bedrock Agents API for autonomous agents
3. `docs/source/02-Use-Case-Catalog/index.rst` — the full spec is already written

### 6. Observability and ops readiness

**Current state**: No structured logging, no metrics, no tracing. `print()` statements in example scripts. No cost tracking per run.

**Senior-engineer target state**: Structured JSON logs with `run_id`, `use_case_id`, `prompt_id`, `prompt_version`, `model_id`, `latency_ms`, `input_token_count`, `output_token_count`, `eval_result`. A cost estimator that tracks token usage per evaluation run. Optional: OpenTelemetry spans for each Bedrock call.

**Alternatives**:
- *LangSmith / Weights & Biases* — third-party LLM observability platforms. Pro: UI, dashboards, comparison across runs. Con: external dependency, data leaves the org, cost.
- *Structured logging to S3* — write JSONL evaluation logs to S3. Pro: durable, queryable via Athena. Con: setup overhead.

**Knowledge prerequisites**: Python `logging` module, OpenTelemetry basics, AWS Bedrock token usage fields in the Converse API response.

**Suggested learning path**:
1. Python `logging` cookbook (structured logging with `extra`)
2. Bedrock Converse API response schema (`usage` field)
3. OpenTelemetry Python quick-start

### 7. Developer experience (DX)

**Current state**: New prompts require touching 5 files manually (template, enum, runner, example, test data). The `gen-prompt` skill automates this, but it requires Claude Code. No type-checked `mypy` gate. No linter/formatter config.

**Senior-engineer target state**: `mypy` configured and run in CI (`pyproject.toml [tool.mypy]`). `ruff` as linter + formatter (replaces separate `flake8`/`black`). The `gen-prompt` skill should have a corresponding `mise run gen-prompt <args>` task.

**Alternatives**:
- *Full `mypy --strict`* — aggressive type checking catches more bugs but requires annotating everything. Pro: catches subtle bugs early. Con: adds annotation burden.
- *`pyright`* — Microsoft's type checker, faster than mypy. Pro: better VS Code integration. Con: different behavior on edge cases.

**Knowledge prerequisites**: `mypy` configuration, `ruff` rule sets, `pyproject.toml` `[tool.*]` sections.

**Suggested learning path**:
1. `mypy` docs: `pyproject.toml` configuration
2. `ruff` docs: rule selection
3. Astral's `uv` integration with ruff
