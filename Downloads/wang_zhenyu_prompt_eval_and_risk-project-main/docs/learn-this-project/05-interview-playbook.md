# 05 — Interview Playbook

> For the interviewer skill: never show model answers to the user mid-interview. Use these as grading references and for debrief.

## Opening — the 90-second pitch (model script)

> `prompt_risk` is a Python framework I built for detecting security vulnerabilities in LLM prompts at enterprise scale. The core insight is that traditional security scanning — regex, keyword lists — can't catch semantic threats like over-permissive authorization or role confusion. So the framework combines two layers: fast deterministic checks for obvious risks, and an LLM-as-Judge pipeline for semantic analysis.
>
> I implemented it in the context of insurance AI, using six real use cases ranging from a claims intake pipeline to autonomous agents. For each use case, I built versioned prompt templates, normal and adversarial test datasets in TOML format, and automated evaluation pipelines that check both business correctness and injection resistance.
>
> The current implementation has the full claim intake pipeline working end-to-end — three extraction steps with Pydantic output validation and retry-on-failure — plus the first security judge (J1, over-permissive authorization) that scores prompts across five criteria. The system can intentionally load vulnerable prompt versions and demonstrate how J1 catches them.

## Round 1 — What you have

**Q1.1**: "Walk me through what happens when I call `run_p1_extraction()` with a normal test case — from the moment the function is called to the moment it returns."

*Model answer*: The runner instantiates `Prompt(id=PromptIdEnum.UC1_P1_EXTRACTION, version=...)` which lazy-loads `system-prompt.jinja` and `user-prompt.jinja` from `data/uc1-claim-intake/prompts/p1-extraction/versions/{ver}/`. It renders the user prompt with the input data via Jinja2. It calls `bedrock_utils.converse()` with a system list (text + cachePoint) and a messages list. The Bedrock API returns text; `extract_json()` strips any markdown fence and parses the JSON. Pydantic validates the dict against `P1ExtractionOutput`; if valid, returns. If invalid, the error is appended to conversation history and the loop retries (up to MAX_RETRIES=3).

*Follow-ups*: "What exactly gets appended to conversation history on retry?" / "Why is the system prompt in a list with a cachePoint entry?"

*Weak answer signature*: Says "it calls the LLM and parses the output" without mentioning retry, caching, or Jinja rendering.

---

**Q1.2**: "How does the assertion-based evaluation work for attack test cases?"

*Model answer*: Attack test cases have an `[attack_target]` section in their TOML file specifying field-value pairs that an attacker wants the LLM to produce (e.g., `injury_indicator = "none"` after injecting instructions). `evaluate()` runs `ne` assertions — the output must NOT equal those values. A passing attack test means the prompt resisted the injection.

*Follow-ups*: "What if both `[expected]` and `[attack_target]` are present in the same TOML?" / "What does it mean if an attack test passes?"

*Weak answer signature*: Confuses `ne` with checking the attack content in the input rather than the output.

---

**Q1.3**: "What is the J1 judge actually evaluating — the prompt's runtime behavior, or the prompt text itself?"

*Model answer*: The prompt text itself — specifically the system and user prompt templates. J1 reads the Jinja templates and evaluates whether they are semantically over-permissive (e.g., "never refuse", unconstrained scope, no injection guardrails). It does NOT execute the prompt or observe its behavior. This means it can catch vulnerabilities before any test case is run.

*Follow-ups*: "What are the 5 criteria?" / "How would J1's approach miss a vulnerability?"

*Weak answer signature*: Says J1 observes the LLM's outputs — confuses static analysis with dynamic evaluation.

---

**Q1.4**: "Why does `P1LoaderEnum` load all test cases at import time rather than lazily?"

*Model answer*: WHY (best guess — verify with project owner): convenience. Developers see the full list of available test cases when they import the module. Import-time errors (missing TOML, malformed TOML) surface immediately rather than at runtime. The downside is that adding a new TOML file without updating the enum causes a silent gap; deleting a TOML file causes an import error.

*Follow-ups*: "What's the trade-off vs. lazy loading?" / "How would you change this for a test suite with hundreds of cases?"

## Round 2 — What you'd elevate

**Q2.1**: "The J1 judge exists, but J2–J5 don't. What would you prioritize next, and why?"

*Model answer*: J2 (sensitive data / secrets in prompts) is the highest-value next judge because it has the most mature tooling to borrow from (detect-secrets, trufflehog) and the risk is easy to demonstrate. Alternatively, J3 (role confusion) because the attack surface grows as the framework adds customer-facing use cases (UC6). In either case, the meta-judge aggregator should be scaffolded once at least 2 judges exist — the aggregation logic determines how individual judge scores translate to policy decisions.

*Follow-ups*: "Why not J2 as a deterministic rule (regex) rather than an LLM judge?" / "What does the meta-judge do that individual judges can't?"

---

**Q2.2**: "The AWS profile is hardcoded as `wang_zhenyu_dev`. What would you do about that if a second developer joined?"

*Model answer*: Replace the hardcoded profile with a credential chain: (1) `AWS_PROFILE` env var → (2) `AWS_ACCESS_KEY_ID` + `AWS_SECRET_ACCESS_KEY` → (3) instance metadata. Alternatively, since runners already accept `client` as a parameter, fully remove the `one` singleton and require callers to inject their own client — the dependency injection approach. For CI, use GitHub Actions OIDC to get short-lived IAM tokens with no stored secrets.

*Follow-ups*: "Would you keep the `one` singleton at all?" / "What's the risk of the current approach in CI?"

---

**Q2.3**: "The `tests_int/` directory is empty. What would you put in it?"

*Model answer*: At minimum: (1) one integration test per implemented runner that calls real Bedrock, verifies the output passes Pydantic validation, and asserts sensible field values. (2) Known-answer tests for J1: run J1 on v01 (known-good) and v02 (known-critical) and assert the score is ≥4 for v01 and ≤2 for v02. (3) Gate them with `@pytest.mark.integration` so they skip in normal CI runs, triggered only when `RUN_INTEGRATION_TESTS=1`.

*Follow-ups*: "How do you prevent these tests from becoming expensive to run?" / "What's the difference between using botocore stubbers vs. real Bedrock calls?"

## Round 3 — Alternatives considered

**Q3.1**: "Why use Jinja2 for prompts rather than Python f-strings?"

*Model answer*: Jinja2 templates live on disk as files, not in Python source. Prompt engineers can edit them without touching Python code or going through a code review aimed at developers. With f-strings, every prompt change requires a Python file edit and potentially breaks syntax or runtime behavior. Jinja2 also enables looping and conditionals in templates if needed, though the current prompts use simple variable substitution.

*Follow-ups*: "What if the prompt engineer accidentally introduces a Jinja2 injection in the template itself?" / "Why not use LangChain's PromptTemplate?"

---

**Q3.2**: "Why roll your own `extract_json()` instead of using a library like `instructor` or `outlines` for structured output?"

*Model answer*: WHY (best guess — verify with project owner): the project aims to be a lightweight reference framework. `instructor` and `outlines` add nontrivial dependencies and opinions about how to structure LLM calls. Rolling `extract_json()` means the extraction logic is fully transparent, easily audited, and doesn't pull in a framework that might change its API. The retry-on-validation feedback loop is also hand-rolled for the same reason — it's a teaching artifact.

*Follow-ups*: "What would `instructor` have given you that your approach lacks?" / "At what scale would you switch to a library?"

---

**Q3.3**: "Why store test data as TOML rather than Python dicts or JSON?"

*Model answer*: TOML has native multiline string support (`"""..."""`) which is critical for FNOL narratives that span multiple paragraphs. JSON requires escaped newlines. Python dicts are Python — non-engineers can't review or edit them without understanding Python syntax. TOML is diff-friendly and human-readable, making it suitable for a security team to review adversarial test cases without a developer intermediary.

*Follow-ups*: "What's the downside of TOML vs. YAML?" / "How do you validate that a new TOML file is well-formed before merging?"

## Round 4 — Problems hit

**Q4.1**: "What happens when the LLM returns the date in a format like `April 15, 2026` instead of `2026-04-15`? Walk me through what the system does."

*Model answer*: The Pydantic `@field_validator` on `date_of_loss` rejects the format and raises a `ValidationError`. The runner catches it, appends the error message plus the raw response to conversation history, and calls Bedrock again. The LLM sees the error "field date_of_loss must be YYYY-MM-DD format" and (usually) self-corrects on retry 2. If it fails 3 times, `ValidationError` propagates.

*Follow-ups*: "How would you improve this so it doesn't cost 3 API calls for a predictable format issue?" / "What if the date is genuinely ambiguous in the narrative?"

---

**Q4.2**: "If you ran `run_p2_classification()` and got a wrong `line_of_business` classification, how would you debug it?"

*Model answer*: First check whether the P1 output fed to P2 is correct — P2 doesn't re-read the narrative, only P1's JSON. If P1 is wrong, fix P1. If P1 is right, inspect the P2 system prompt to see if the LOB definitions are clear and non-overlapping. Run the P2 example script with different test cases to isolate whether it's one LOB that fails or all. Check the `reasoning` field in `P2ClassificationOutput` — the LLM explains its classification there.

*Follow-ups*: "What would you add to `P2ClassificationOutput` to make debugging easier?" / "How would you test that a fix to P2's system prompt didn't break any previously passing cases?"

## Round 5 — Production pushbacks

**Q5.1**: "If traffic to this system 100x'd overnight — suddenly evaluating thousands of prompts per day — what breaks first?"

*Model answer*: (1) Bedrock rate limits — the current code has no rate limiting, backoff, or request queuing. (2) Cost — no token budget or cost circuit breaker. (3) The `one` singleton is not thread-safe if judges run in parallel threads (boto3 clients ARE thread-safe but the lazy-init pattern may have a race condition). (4) No async support — everything is synchronous. At scale, async Bedrock calls + a work queue (SQS + Lambda, or Celery) would be needed.

*Follow-ups*: "Would you keep Bedrock, or switch to a different LLM provider at scale?" / "How would you detect if Bedrock is throttling you?"

---

**Q5.2**: "A prompt version that passed J1 with score 4 (LOW) in January now fails with score 1 (CRITICAL) in March. How do you investigate?"

*Model answer*: J1 itself is an LLM call — the judge's behavior may have changed due to a model update (LLM version drift). First check if the same judge version (`j1/versions/01`) produces consistent results across 5+ runs (stochastic variance). Then compare: did the target prompt change? (git blame on the `.jinja` file). Did the judge prompt change? Did the underlying model change? The core fix is to add judge-quality tests (known-answer tests for J1 on its own known-good and known-bad prompts) to CI, so regressions are caught before they reach production.

*Follow-ups*: "How do you distinguish a real regression from LLM stochasticity?" / "What would a `judge-quality` CI job look like?"

---

**Q5.3**: "You're demoing this to a security team that asks: 'How do we know J1 isn't hallucinating findings?' What do you say?"

*Model answer*: J1 is validated against known-answer test cases — v01 (production quality) should consistently score ≥4, v02 (over-permissive) should consistently score 1. The per-criterion findings include evidence quotes from the prompt text, so a human can verify whether the cited text actually supports the finding. The multi-version comparison (same prompt across multiple judge runs) establishes statistical confidence. The full framework spec describes cross-model comparison (run J1 on multiple LLMs and check for agreement) as a quality assurance step — though this isn't automated yet.

---

**Q5.4**: "What happens if Bedrock retires the model you're using (`us.amazon.nova-2-lite-v1:0`)?"

*Model answer*: Every runner and judge accepts `model_id` as a parameter — it's not hardcoded in the function signatures. The default value in each runner is the current model string. To migrate, update the defaults in each runner. A better design would centralize the default `model_id` in one place (`constants.py` or an env var) so it's a one-line change. Currently there's no test that would catch a broken model ID until someone runs an example.
