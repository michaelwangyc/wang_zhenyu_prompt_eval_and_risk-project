- **Q-001** [tag: knowhow] [difficulty: easy]
  - **Q:** What two layers of defense does `prompt_risk` combine, and why does it use both?
  - **A:** Deterministic rule scanning (R1/R2 — secrets, keywords) and LLM-as-Judge semantic analysis (J1–J5). Rules are fast and cheap but miss semantic threats like over-permissive authorization; LLM judges catch what regex can't.
  - **Source:** `docs/learn-this-project/01-knowhow-inventory.md#project-at-a-glance`

- **Q-002** [tag: knowhow] [difficulty: easy]

  - **A:** `"{use_case_id}:{prompt_short_name}"`, e.g. `"uc1-claim-intake:p1-extraction"`. It encodes the use case and prompt so the `.dir_root` property can locate the `data/` directory without any extra argument.

- **Q-005** [tag: knowhow] [difficulty: medium]
  - **Q:** Why does `path_enum` use absolute paths instead of paths relative to `__file__`?
  - **A:** Absolute paths are CI/CD safe — the working directory doesn't matter. Paths relative to `__file__` break when the package is installed into a virtualenv (not editable) or when tests change directory.
  - **Source:** `prompt_risk/paths.py`, `docs/learn-this-project/01-knowhow-inventory.md`

- **Q-006** [tag: knowhow] [difficulty: medium]
  - **Q:** What is the purpose of `[attack_target]` in a test case TOML file?
  - **A:** It specifies field → forbidden value pairs. The `evaluate()` function checks that the LLM's output does NOT equal those values (`ne` assertion), verifying the prompt resisted a prompt injection attack.
  - **Source:** `prompt_risk/evaluations.py`, `data/uc1-claim-intake/prompts/p1-extraction/attack/`

- **Q-007** [tag: knowhow] [difficulty: medium]
  - **Q:** What happens after a Pydantic `ValidationError` in a runner's retry loop?
  - **A:** The validation error message and the LLM's raw response are both appended to the conversation history as a new user message, then `converse()` is called again. This gives the LLM concrete feedback to self-correct rather than retrying blindly.
  - **Source:** `prompt_risk/uc/uc1/p1_extraction_runner.py`

- **Q-008** [tag: knowhow] [difficulty: medium]
  - **Q:** What is `MAX_RETRIES` in the runner pattern, and what happens when it's exhausted?
  - **A:** `MAX_RETRIES = 3`. After 3 failed validation attempts, the runner raises the `ValidationError` (or `JsonExtractionError`) uncaught, propagating to the caller.
  - **Source:** `prompt_risk/uc/uc1/p1_extraction_runner.py:MAX_RETRIES`

- **Q-024** [tag: knowhow] [difficulty: hard]
  - **Q:** Why does `Prompt` use `functools.cached_property` for template loading, and what's the gotcha?
  - **A:** `cached_property` avoids re-reading the file on every render call within the same `Prompt` instance. The gotcha: if you modify the `.jinja` file on disk, a reused instance still serves the old template. Always create a new `Prompt` instance to get fresh templates.
  - **Source:** `prompt_risk/prompts.py`

- **Q-065** [tag: knowhow] [difficulty: hard]
  - **Q:** Why is the extraction judge's input structured as `(original input, LLM output)` rather than just `(LLM output)`?
  - **A:** Without the original input, the judge can't verify factual accuracy — it can only check schema validity. For "is `date_of_loss` correct?", the judge must compare the extracted date against what was mentioned in the narrative. The original input is the ground truth.
  - **Source:** `.claude/skills/gen-judge/SKILL.md`, `prompt_risk/uc/uc1/p1_extraction_judge_runner.py`

- **Q-070** [tag: knowhow] [difficulty: hard]
  - **Q:** What would happen if `P1LoaderEnum.A_01_INJECTION_IN_NARRATIVE` was run through P1 with prompt version `v02` (over-permissive), and J1 also evaluated `v02`?
  - **A:** P1 with `v02` would likely produce output matching the attacker's injected values (the attack succeeds — business failure + security failure). J1 would score `v02` as CRITICAL (score 1) because the prompt lacks refusal capability. This demonstrates the diagnostic matrix: the same prompt fails both business and security evaluation.
