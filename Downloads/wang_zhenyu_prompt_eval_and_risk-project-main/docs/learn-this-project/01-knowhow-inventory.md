# 01 — Knowhow Inventory

## Project at a glance

`prompt_risk` is a Python framework for detecting, scoring, and mitigating security risks in LLM prompts deployed across enterprise environments. It combines **deterministic rule-based scanning** (secrets detection, keyword blocklists — R1/R2) with **LLM-as-Judge semantic analysis** (J1–J5 judges) to catch vulnerabilities that regex alone cannot find: over-permissive authorization, hardcoded sensitive data, role confusion, instruction conflicts, and logic ambiguity.

The project ships six insurance-industry use cases as reference implementations, each with versioned Jinja prompt templates, normal and adversarial TOML test cases, and automated evaluation pipelines. Currently UC1 (claim intake, P1–P3) and J1 (over-permissive authorization) are implemented; the rest are specified but planned.

Primary users: security engineers and ML platform teams who want prompt security as a repeatable CI/CD gate, not a manual ad-hoc review.

## Component inventory

### `prompt_risk/constants.py`

- **What**: Enumerates all use case IDs (`UseCaseIdEnum`) and prompt IDs (`PromptIdEnum`). Each `PromptIdEnum` member has a computed `.dir_root` property that returns the absolute `data/{use_case_id}/prompts/{prompt_short_name}/` directory.
- **Why**: A single authoritative enum prevents magic strings from scattering across the codebase. Any new prompt is registered here first; mismatches between code and filesystem surface at import time rather than at runtime.
- **Key files**: `prompt_risk/constants.py`
- **Depends on**: `prompt_risk/paths.py`
- **Gotchas**: The enum value format is `"{use_case_id}:{prompt_short_name}"` (colon-separated). `dir_root` splits on `:` to recover both parts. Don't add members with a different separator.

---

### `prompt_risk/paths.py`

- **What**: A singleton `path_enum` that exposes all project-root-relative absolute paths (`dir_project_root`, `dir_data`, `dir_htmlcov`, etc.) as `Path` objects.
- **Why**: Absolute paths are CI/CD safe — no matter what the working directory is when pytest or sphinx runs, paths resolve correctly. Alternative (relative paths from `__file__`) breaks when installed as a package or when tests change directory.
- **Key files**: `prompt_risk/paths.py`
- **Depends on**: Python stdlib (`pathlib`)
- **Gotchas**: `dir_project_root` is computed via `Path(__file__).parent.parent` — it points to the repo root, not the package root. If the package is ever installed into a virtualenv (not editable), this computation breaks.

---

### `prompt_risk/prompts.py`

- **What**: A `Prompt` dataclass that lazy-loads and caches the Jinja2 system and user prompt templates for a given `(id, version)` pair. Exposes `.system_prompt_template` and `.user_prompt_template` as `jinja2.Template` objects.
- **Why**: Centralizes the "find the right file on disk and parse it" logic. Every runner instantiates `Prompt(id=..., version=...)` and renders without knowing the file layout.
- **Key files**: `prompt_risk/prompts.py`
- **Depends on**: `jinja2`, `constants.py` (for `dir_root`), `paths.py`
- **Gotchas**: Templates are cached via `functools.cached_property`. If you modify a template file and the `Prompt` instance is reused, the old template is served. Always create a new `Prompt` instance per run (which runners do).

---

### `prompt_risk/bedrock_utils.py`

- **What**: `converse(client, model_id, system, messages) → str` — a thin wrapper around the AWS Bedrock Converse API. Accepts a system prompt list (which may include `cachePoint` entries) and a messages list; returns the assistant's text response.
- **Why**: Isolates the Bedrock SDK call behind a simple interface. Runners don't need to know about response nesting (`response["output"]["message"]["content"][0]["text"]`).
- **Key files**: `prompt_risk/bedrock_utils.py`
- **Depends on**: `boto3` (Bedrock Runtime client, passed in — not created here)
- **Gotchas**: The function assumes the first content block is a text block. Multi-modal responses (images, tool_use) are not handled and would silently return the wrong content.

---

### `prompt_risk/llm_output.py`

- **What**: `extract_json(text: str) → dict` — extracts a JSON object from LLM output that may be wrapped in markdown fences (` ```json ... ``` ` or ` ``` ... ``` `) or returned as bare text.
- **Why**: LLMs often wrap JSON in markdown even when instructed not to. Rather than instructing the model repeatedly, the post-processor strips the fence unconditionally. The "first fence wins" rule handles models that include both a json fence and a trailing plain fence.
- **Key files**: `prompt_risk/llm_output.py`
- **Depends on**: Python stdlib (`json`, `re`)
- **Gotchas**: Raises `JsonExtractionError` (not `json.JSONDecodeError`) on failure. The error includes the full raw text for debugging but also a truncated 200-char preview for display. Arrays (`[...]`) are also handled — not just objects.

---

### `prompt_risk/exc.py`

- **What**: `JsonExtractionError` — a custom exception that wraps LLM parse failures. Stores the full raw output in `.raw_output` and a 200-char truncated preview in the message.
- **Why**: Structured exceptions make it easy to log the raw LLM output for debugging without dumping kilobytes to stderr. The `MAX_PREVIEW_LENGTH = 200` constant is a deliberate cap.
- **Key files**: `prompt_risk/exc.py`
- **Depends on**: nothing
- **Gotchas**: None.

---

### `prompt_risk/evaluations.py`

- **What**: `evaluate(output: BaseModel, expected: dict | None, attack_target: dict | None) → EvalResult`. Runs assertions against an LLM output model using three operators: `eq` (exact match), `in` (value in list), `ne` (injection-resistance: output must NOT equal the target). Returns `EvalResult` with `.passed` (bool), `.field_results` (per-field breakdown), and `.score` (fraction correct).
- **Why**: Separates **business correctness** (did the model extract the right facts? — `expected`) from **security** (did the prompt resist injection? — `attack_target`). This enables a 2×2 diagnostic matrix: a prompt can pass business tests but fail security, or vice versa.
- **Key files**: `prompt_risk/evaluations.py`
- **Depends on**: `pydantic`
- **Gotchas**: `ne` assertions check that the output field does NOT equal the attack target's value. A pass on `ne` means the injection was resisted. The field value is accessed via `getattr(output, field_name)` — field names must match the Pydantic model exactly.

---

### `prompt_risk/one/` (AWS credential singleton)

- **What**: A mixin-based singleton `one` (instance of `One`) that lazily initializes a boto3 session and Bedrock Runtime client. `one.bedrock_runtime_client` is the entry point for all examples.
- **Why (best guess; verify with project owner)**: Centralizes credential management so examples don't repeat `boto3.Session(profile_name=...)` boilerplate. The mixin pattern (`OneBotoSesMixin`, `OneConfigMixin`) allows adding more AWS services (S3, DynamoDB) without modifying the base class.
- **Key files**: `prompt_risk/one/api.py` (re-exports `one`), `prompt_risk/one/one_01_main.py`, `prompt_risk/one/one_03_boto_ses.py`
- **Depends on**: `boto3`
- **Gotchas**: AWS profile is hardcoded as `"wang_zhenyu_dev"` in `one_03_boto_ses.py`. This means examples only work if the developer has this profile configured. CI/CD would need to either mock this or use environment-variable credentials. `one_02_config.py` has a `config` property that returns `None` — it's a stub.

---

### `prompt_risk/judges/j1_over_permissive.py`

- **What**: The J1 "over-permissive authorization" judge. `run_j1(client, target_prompt_text, user_data, judge_version, model_id) → J1Result`. Evaluates a target prompt across 5 criteria: (1) refusal capability, (2) scope boundaries, (3) unconditional compliance, (4) failure handling, (5) anti-injection guardrails. Returns per-criterion findings with severity (pass/minor/major), evidence strings, and recommendations, plus an overall `score` (1–5) and `overall_risk` (pass/low/medium/high/critical).
- **Why**: Security properties of a prompt text cannot be checked by regex (e.g., "does the system prompt allow the LLM to be instructed to ignore safety rules?"). LLM-as-Judge performs semantic analysis. J1 is the first of five planned judges; it evaluates the most critical risk (over-permissive authorization).
- **Key files**: `prompt_risk/judges/j1_over_permissive.py`
- **Depends on**: `prompts.py`, `bedrock_utils.py`, `llm_output.py`, `pydantic`
- **Gotchas**: The judge evaluates **prompt text**, not LLM behavior. It reads the system + user prompt templates, not a live conversation. `print_j1_result()` is a utility for human-readable terminal output with emoji risk icons.

---

### `prompt_risk/uc/uc1/p1_extraction_runner.py`

- **What**: `run_p1_extraction(client, data: P1ExtractionUserPromptData, prompt_version, model_id) → P1ExtractionOutput`. Runs the P1 FNOL extraction prompt: given a raw insurance claim narrative, extracts 10 structured fields (date_of_loss, time_of_loss, location, line_of_business_hint, parties_involved, damage_description, injury_indicator, police_report, evidence_available, estimated_severity). Uses system-prompt caching and retry-on-validation-failure (MAX_RETRIES=3).
- **Why**: P1 is the entry point of the UC1 pipeline. All downstream steps (P2 classification, P3 triage) consume P1's JSON output. Caching the static system prompt reduces cost on repeated calls.
- **Key files**: `prompt_risk/uc/uc1/p1_extraction_runner.py`
- **Depends on**: `prompts.py`, `bedrock_utils.py`, `llm_output.py`, `pydantic`
- **Gotchas**: `date_of_loss` has a `@field_validator` enforcing `YYYY-MM-DD` format. If the LLM returns a different format, validation fails and triggers a retry with the error appended to conversation history.

---

### `prompt_risk/uc/uc1/p1_test_data.py`

- **What**: `P1ExtractionUserPromptDataLoader` — loads a single TOML test case file and exposes `.data` (the `P1ExtractionUserPromptData` ready to pass to the runner), `.expected` (dict of field → expected value for `eq`/`in` assertions), and `.attack_target` (dict of field → forbidden value for `ne` assertions). `P1LoaderEnum` enumerates all 9 test cases (6 normal `b-*`, 3 attack `a-*`).
- **Why**: Decouples test data from runner code. TOML files are human-reviewable, diff-friendly, and allow security team members to add test cases without writing Python.
- **Key files**: `prompt_risk/uc/uc1/p1_test_data.py`, `data/uc1-claim-intake/prompts/p1-extraction/normal/`, `data/uc1-claim-intake/prompts/p1-extraction/attack/`
- **Depends on**: `tomllib` (Python 3.11 stdlib), `pydantic`
- **Gotchas**: The `P1LoaderEnum` values are pre-instantiated `P1ExtractionUserPromptDataLoader` objects (not lazy). All 9 TOML files are read at import time.

---

### `prompt_risk/uc/uc1/p1_extraction_judge_runner.py`

- **What**: `run_p1_extraction_judge(client, data: P1ExtractionJudgeUserPromptData, prompt_version, model_id) → P1ExtractionJudgeOutput`. A **business-correctness** judge that takes the original FNOL narrative (input) and the P1 extraction output (output) and evaluates whether every extracted field is factually correct. Returns `pass_` (bool), `reason` (str), `field_errors` (list of `{field, issue}`).
- **Why**: Assertion-based evaluation (`evaluations.py`) can only check a few key fields with hard-coded expected values. The extraction judge covers ALL fields, including free-text fields like `damage_description` that can't be compared with `eq`. This is distinct from J1 (security) — the extraction judge checks business correctness only.
- **Key files**: `prompt_risk/uc/uc1/p1_extraction_judge_runner.py`
- **Depends on**: `prompts.py`, `bedrock_utils.py`, `llm_output.py`, `pydantic`
- **Gotchas**: The Pydantic field is named `pass_` (with underscore) but serialized as `"pass"` via `Field(alias="pass")` — a Python reserved-word workaround.

---

### `prompt_risk/uc/uc1/p2_classification_runner.py`

- **What**: `run_p2_classification(client, data: P2ClassificationUserPromptData, prompt_version, model_id) → P2ClassificationOutput`. Classifies the line of business from P1 JSON output. Output: `line_of_business` (auto/property/workers_comp/general_liability/marine/cyber/other), `confidence` (high/medium/low), `reasoning`, `secondary_lob`, `field_conflicts`, `escalate`.
- **Why**: Separating extraction (P1) from classification (P2) allows each step to be improved, versioned, and evaluated independently. A classification error is diagnosable without rerunning extraction.
- **Key files**: `prompt_risk/uc/uc1/p2_classification_runner.py`
- **Depends on**: same pattern as P1
- **Gotchas**: P2 takes P1 output as input; it doesn't re-read the original narrative. This means a P1 error propagates silently to P2.

---

### `prompt_risk/uc/uc1/p3_triage_runner.py`

- **What**: `run_p3_triage(client, data: P3TriageUserPromptData, prompt_version, model_id) → P3TriageOutput`. Takes both P1 and P2 output, assigns `severity_level` (1–5) and `handling_priority` (urgent/high/standard/low). Also returns `reasoning`, `field_conflicts`, `escalate`.
- **Why**: Triage is downstream of both extraction and classification — it needs the full picture to assign priority correctly.
- **Key files**: `prompt_risk/uc/uc1/p3_triage_runner.py`
- **Depends on**: same pattern as P1/P2
- **Gotchas**: P4 (coverage check) and P5 (routing) are planned but not yet implemented. Their `data/` directories exist but are empty.

---

### `data/` directory (test data and prompt templates)

- **What**: The declarative layer. Contains Jinja2 prompt templates (`system-prompt.jinja`, `user-prompt.jinja`) and TOML test cases for every implemented use case and judge. Layout: `data/{use_case_id}/prompts/{prompt_short_name}/versions/{version}/` for templates; `data/{use_case_id}/prompts/{prompt_short_name}/normal/` and `attack/` for test cases.
- **Why**: Separating data from code enables non-engineers to review and modify prompts and test cases via PR. The version directory allows multiple prompt variants to coexist without branching code.
- **Key files**: `data/uc1-claim-intake/prompts/p1-extraction/versions/01..04/`, `data/judges/prompts/j1-over-permissive/versions/01/`
- **Gotchas**: Normal test cases use `b-{nn}-{name}.toml` naming; attack cases use `a-{nn}-{name}.toml`. The `b`/`a` prefix is cosmetic — the runner uses `P1LoaderEnum` to find files, not a glob.

---

### `examples/` (demo and manual testing scripts)

- **What**: Standalone Python scripts that exercise the framework end-to-end. Each file loads a test case enum, runs the relevant runner, evaluates the output, and prints results. A commented block at the top of each file lists all available test cases — uncomment one to run it.
- **Why**: `examples/` scripts serve as interactive development tools AND as live documentation — they show the exact call signature of every runner.
- **Key files**: `examples/uc1-claim-intake/run_uc1_p1_extraction.py`, `examples/judges/run_j1_on_uc1_p1.py`
- **Gotchas**: These require live AWS credentials (via `one.bedrock_runtime_client`) and incur Bedrock API costs. They are NOT run in CI.

---

### `tests/` (unit tests)

- **What**: Four pytest modules covering the pure-logic layer: `test_evaluations.py` (25+ cases for `evaluate()`), `test_llm_output.py` (13+ cases for `extract_json()`), `test_prompts.py` (reads and renders two prompt templates), `test_api.py` (smoke test for the `one` API). Each test file is also runnable as `python tests/test_*.py` via the `run_cov_test()` helper.
- **Why**: Unit tests cover the deterministic parts only — no LLM calls, no AWS. They run fast, offline, and in CI without credentials.
- **Key files**: `tests/test_evaluations.py`, `tests/test_llm_output.py`, `tests/test_prompts.py`
- **Depends on**: `pytest`, `pytest-cov`, `prompt_risk/vendor/pytest_cov_helper.py`
- **Gotchas**: No integration tests (yet). `tests_int/` and `tests_load/` directories exist but are empty.

## Conventions

**Naming**:
- Use case IDs: kebab-case, e.g. `uc1-claim-intake`
- Prompt short names: kebab-case, e.g. `p1-extraction`
- Python module names: snake-case equivalent, e.g. `p1_extraction_runner`
- Enum members: `UC1_P1_EXTRACTION` (upper snake, joining use case + prompt)
- Test cases: `b-{nn}` for normal, `a-{nn}` for attack

**Runner pattern** (every runner follows this exactly):
1. Instantiate `Prompt(id=..., version=...)`.
2. Build `system` list with `cachePoint` after system prompt text.
3. Render `user_prompt` from `data`.
4. Call `converse()` in a retry loop (max 3 attempts).
5. On validation failure, append the error as a new user message, retry.
6. Return validated `*Output` Pydantic model.

**Test data TOML structure**:
```toml
[meta]       # id, description, line_of_business, complexity, date_created
[input]      # fields consumed by user-prompt.jinja (e.g. source, narrative)
[expected]   # field → expected value (for eq/in assertions on normal cases)
[attack_target]  # field → forbidden value (for ne assertions on attack cases)
```

**Error handling style**: Fail fast and loud. `extract_json()` raises `JsonExtractionError` with the raw output attached. Runners raise `ValidationError` (Pydantic) after MAX_RETRIES. No silent fallbacks.

## External dependencies (non-obvious choices)

| Dependency | Used for | Why this, not alternatives |
|------------|----------|---------------------------|
| `pydantic v2` | Output validation + retry feedback | Field-level error messages fed back to the LLM are much cleaner than raw `dict` diffs. `v2` is required for `model_fields` introspection used in test data loading. |
| `jinja2` | Prompt templates | Enables `{{ data.field }}` variable substitution without building a custom parser. Chosen over f-strings because templates live on disk and are owned by prompt engineers, not developers. |
| `boto3` | AWS Bedrock calls | Bedrock is the hosting platform for the demo models (Amazon Nova). Not a general choice — tied to the project's AWS setup. |
| `claude-agent-sdk` | (Required but usage unclear in current codebase; WHY: best guess: future agent orchestration layer — verify with project owner) | |
| `rich` | Pretty terminal output | Available but lightly used. `print_j1_result()` in `j1_over_permissive.py` uses it for colored risk level display. |
| `sphinx` + `furo` + `nbsphinx` | Documentation build | `nbsphinx` is the key dependency — it allows Jupyter notebooks (`.ipynb`) to be compiled into HTML pages alongside `.rst` docs. |
